## Introdução

As métricas comumente utilizadas para avaliar a qualidade de um ou mais assemblies podem ser resumidas como os “3C” (Contiguity, Completeness e Correctness). 

Resumidamente, Contiguity é frequentemente medido como contig N50, que é o comprimento de corte para os contigs mais longos que contêm 50% do comprimento total do genoma. Um contig N50 acima de 1 Mb é geralmente considerado bom.  Completeness, por sua vez, refere-se à presença ou ausência de genes altamente conservados em uma montagem. O objetivo é ter o maior número de genes identificados em sua montagem. Uma pontuação acima de 95% é considerada boa. Por fim, correctness pode ser definida como a precisão de cada par de bases na montagem, frequentemente medida como a concordância de uma montagem com uma referência de alto padrão de qualidade (quando existente). Além disso, é importante ressaltar que a medição da concordância dos dados só será acurada quando houver baixa divergência entre a amostra de referência e a amostra a ser analisada. Todas essas informações - e mais detalhes - podem ser encontradas aqui). 

Nesta prática, faremos algumas análises rápidas que nos fornecem informações sobre Contiguity e Completeness, visto a maior praticidade em termos de tempo e de recursos computacionais.

O BUSCO (Benchmarking Universal Single-Copy Orthologs) é uma ferramenta utilizada para a determinação quantitativa da completude de assemblies. Esse programa parte do arquivo das sequências nucleotídicas e da escolha de um banco de dados (dataset) com genes característicos de distintas linhagens, com distintos níveis taxonômicos (de famílias à domínios). O BUSCO busca no assembly, a porcentagem de genes completos encontrados (a partir do dataset), se estão duplicados ou em cópia única, genes fragmentados e a porcentagem de genes que não foram encontrados no banco de dados.

## Parte II. Estimativa do tamanho genômico

Nesta etapa, usaremos o genomescope e o jellyfish para fazer estimativas apenas com as reads de um genoma. A vantagem de fazer estimativas com as raw reads é que não precisamos montar o genoma todo para ver o resultado.

Para entender como essas estimativas podem ser feitas, clique aqui e leia o texto. Vamos fazer um teste com um conjunto de reads de Acinetobacter baumannii disponível no NCBI. Essas reads estão no arquivo SRR1945430.fastq.gz, já baixado no início desse tutorial.

Usaremos o programa jellyfish. Instale-o.

```bash
sudo apt install jellyfish

```

Tendo as reads baixadas e o jellyfish instalado, o próximo passo é fazer a análise de count do jellyfish. Nesta etapa ele conta os Kmers de tamanho -m e salva em um arquivo binário.

```bash
jellyfish count -C -m 21 -s 1000000000 -t 4 -o reads.jf <(zcat *.fastq.gz)

```

Na nossa linha de comando temos:

* `-m` tamanho do kmer desejado
* `-s` quantos kmers vão ser contados (o manual sugere que este número seja baseado no tamanho do genoma estimado e cobertura. Para simplificar usaremos 10^9)
* `-t` número de processadores que serão usados
* `-o` nome do arquivo de saída.

O zcat é para ler o arquivo comprimido que está em ‘.gz’, que serve de entrada para o jellyfish através do símbolo de menor ‘<’ . Para ver as opções do jellyfish use.

```bash
man jellyfish

```

ou acesse o manual do programa.

Uma vez terminado, usaremos o jellyfish para criar um histograma a partir do arquivo de saída.

```bash
jellyfish histo -t 10 reads.jf > reads.histo

```

Mais uma vez, -t representa o número de processadores, reads.jf é o arquivo de entrada e reads.histo o histograma.

Concluída essas etapas, acesse o GenomeScope e faça o upload do arquivo (o default das configurações abaixo está como fizemos). Envie e você terá os gráficos prontos.
